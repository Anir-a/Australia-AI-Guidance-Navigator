Enterprise framework

Enterprise Framework for MCP and Agentic AI Governance

Practical guidance for safe enterprise AI adoption, operational control, platform enablement, and assurance where AI agents connect to tools, data sources and business workflows.

View reference architectureApply to enterprise scenariosImplementation reference patterns

Enterprise position

MCP is an interoperability protocol. It standardises how AI applications connect to tools, prompts and resources.
Agentic AI changes the risk model. Systems may act across connected tools and environments, not only produce text.
The governed platform layer is the control point. Catalogue, policy, access, testing, monitoring and evidence operate as shared services.
Principles

Core governance principles

These principles define the minimum operating posture for enterprise MCP and agentic AI adoption.

01

Bounded purpose

Every agent has a named business purpose, approved user group, permitted systems and prohibited actions.

02

Read-only first

New agents begin with retrieval, summarisation or drafting before write actions are considered.

03

Action tiering

Tool calls are classified as read, write, destructive, privileged or high-impact before release.

04

Human control

High-impact and irreversible actions require human approval, rollback and accountable ownership.

Architecture

MCP enterprise architecture and discovery

The enterprise pattern separates the user experience, AI orchestration, MCP protocol connection, governance gateway, approved MCP servers and downstream systems.

MCP overview: Model Context Protocol is an open interoperability protocol that lets AI applications connect to external tools, resources and prompt templates through MCP servers. In an enterprise, MCP should be treated as the connection standard, while the governance gateway and platform controls decide which connections are allowed, monitored and evidenced.

MCP component overview

Host

The AI application or agent environment that presents the experience to users and decides which approved capabilities are available.

Client

The protocol component inside the host that opens and manages the MCP connection to one server at a time.

Server

The governed connector that exposes approved capabilities to the agent. It sits in front of the actual business system.

Tools, resources and prompts

Tools allow actions, resources provide context, and prompts provide reusable interaction templates. Each requires ownership, classification and release control.

Enterprise MCP discovery setup

1

Discover servers

Maintain an enterprise MCP catalogue covering internal, vendor and approved open-source servers.

2

Discover capabilities

Record each server's tools, resources, prompts, action tier, data class and approved caller list.

3

Discover authorization

For protected remote MCP servers, align clients with the correct authorization server and access policy.

4

Register through gateway

Route production MCP traffic through the gateway so identity, policy, HITL, rate limits and logs are enforced.

5

Review lifecycle

Retire or block servers without owner, logs, current review, approved purpose or passing release evidence.

Incident response

Agent incident response model

Agentic systems require an operational response path because failures may involve tool actions, data exposure, incorrect records, workflow disruption or prompt injection in production.

1. Detect

Trigger from SIEM alert, gateway anomaly, user report, support ticket, cost spike, failed evaluation or unexpected tool-call pattern.

2. Contain

Disable the agent, block the MCP tool, revoke token, pause the gateway route, or switch the agent to read-only mode.

3. Preserve evidence

Capture correlation ID, prompt context, tool call, user identity, MCP server, response, timestamp and affected system record.

4. Assess impact

Classify data exposure, wrong action, customer impact, operational disruption, compliance impact and rollback feasibility.

5. Remediate

Reverse records where possible, notify accountable owners, patch prompts/tools, update access policy and retest failed scenarios.

6. Reopen or retire

Return to production only after release gates pass again. Retire the agent or tool where residual risk is not acceptable.

Governance pattern

Governance is distributed across accountable owners, platform teams and assurance functions. The platform layer provides consistent controls.

Owns the governed AI platform layer: agent catalogue, approved MCP/connectors, prompt patterns, release gates, operational telemetry, support model and evidence packs.

FinOps and usage governance

Cost and capacity control

Multi-vendor agentic AI introduces licence, token, API, connector and infrastructure cost. Cost governance should be part of the platform layer, not an afterthought.

Usage visibility

  • Track agent sessions, model calls, tool calls, connector usage and failed calls.
  • Attribute consumption to product, team, environment and business owner.
  • Separate pilot, production and experimental usage.

Cost controls

  • Set quotas and budget alerts for API, token and connector usage.
  • Apply rate limits for expensive or high-volume tools.
  • Require approval for agents that call paid external APIs or high-cost models.

Value review

  • Review active use, repeat use, support demand, quality failures and avoided manual effort.
  • Retire unused agents and duplicated connectors.
  • Prefer shared MCP servers and reusable prompt/tool patterns over one-off builds.
Decision framework

Release gates and default decision path

The release decision is based on capability, action tier, data exposure, operational support and evidence. A gate that fails blocks release until remediated or formally accepted by the accountable risk owner.

1

Classify the use case

Purpose, users, data, systems, autonomy level, business impact and accountable owner.

2

Classify the action tier

Read, write, destructive, privileged or high-impact. The highest tier drives the release gate.

3

Apply release gates

Access, injection, schema, monitoring, approval, support, rollback and FinOps checks.

4

Operate with evidence

Catalogue status, logs, support pathway, incident handling, cost tracking and lifecycle review.

Minimum release gates

  • Named business owner and platform owner recorded.
  • Agent and MCP server registered in the catalogue.
  • Data class and action tier recorded for every tool.
  • Access policy validated against least privilege.
  • Support category and incident owner available before go-live.

Pass/fail thresholds

  • No high-severity prompt injection or tool misuse test remains open.
  • All tool schemas validate expected, boundary and invalid inputs.
  • All destructive, privileged and high-impact actions require human approval.
  • Monitoring confirms correlation ID, user identity, agent identity and tool name for each action.
  • Rollback or containment path is documented for every write-capable tool.

Action-tier release rule

  • Read: release after access, logging and data-boundary checks pass.
  • Write: release after pilot, reversibility and support checks pass.
  • Destructive: release only with approval workflow, rollback and break-glass owner.
  • Privileged: release only with privileged access controls and additional security review.
  • High-impact: release only with accountable executive approval and monitoring review.

Non-release conditions

  • Unregistered MCP server or unmanaged connector.
  • Shared service account with no attributable identity.
  • External or user-generated content used without injection testing.
  • No incident response path for production failure.
  • No method to monitor usage, cost, errors or exceptions.
Operating model

Roadmap and team responsibility

Implementation begins with discovery, then platform setup, read-only pilots and scaled operations.

1. DiscoveryIndicative duration: 2–4 weeks
  • Inventory agents, AI tools and MCP servers.
  • Identify data classes, action tiers and business owners.
  • Confirm platform owner and governance forum.
2. Platform setupIndicative duration: 4–8 weeks
  • Establish catalogue, approved connector path and release gates.
  • Define action tier dictionary and evidence model.
  • Integrate monitoring and support categories.
3. Read-only pilotIndicative duration: 4–6 weeks
  • Start with retrieval, summarisation and drafting scenarios.
  • Test prompt injection, tool boundaries and access controls.
  • Capture adoption, issues and evidence.
4. Scaled operationsIndicative duration: ongoing, reviewed quarterly
  • Introduce controlled write actions with approval gates.
  • Publish patterns, runbooks and platform dashboard.
  • Review exceptions, incidents and lifecycle status.

Where the team comes in

The AI Experience & Platform Enabler Team owns the reusable platform layer and co-ordinates the release path. Security and data partners validate controls. Business owners approve the purpose and risk. Operations run support and incident workflows. Architecture maintains reference patterns.


Steady operating rhythm: platform backlog, fortnightly release review, monthly risk and controls forum, monthly usage and cost review, quarterly evidence and roadmap review.

Reference patterns

Implementation reference patterns

These patterns define how enterprise teams should implement and govern MCP-connected agents.

Pattern 1: Catalogue before connection

No agent or MCP server should be available for enterprise use until it has a catalogue record, owner, purpose, data class and action tier.

Pattern 2: Read before write

Read-only use cases establish data access, quality, monitoring and support pathways before write actions are enabled.

Pattern 3: Gateway before system access

Agent traffic should pass through a governed gateway or control layer before reaching MCP servers and enterprise systems.

Pattern 4: Test adversarial content

Prompt injection and tool misuse tests are required for agents that read user-generated or external content.

Pattern 5: Evidence by design

Logs, approvals, test outcomes and exceptions are captured during normal operation, not reconstructed later.

Pattern 6: Vendor-neutral platform layer

The same catalogue, control gate and evidence model applies across Microsoft, ServiceNow, Salesforce, GitHub, Atlassian, Snowflake and internal systems.

Pattern 7: Incident path before production

Every production agent requires a defined containment, rollback, evidence preservation and re-release path.

Pattern 8: Cost control by design

API, token, licence and connector costs are monitored and attributed as part of normal platform operation.
